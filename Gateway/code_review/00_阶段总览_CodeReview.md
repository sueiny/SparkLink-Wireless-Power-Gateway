## 2. 学习重点：为什么要按阶段 review

网关程序不是单个脚本，而是一个会长期演进的边缘服务。阶段 review 的意义是：

1. 每完成一个阶段，确认“代码做了什么”与“需求边界”一致。
2. 防止把临时测试逻辑误认为最终功能。
3. 记录哪些能力已经可用，哪些只是预留。
4. 给后续接真实 DTU、WS73、Modbus、服务执行、事件上报留下清晰入口。

本项目最重要的边界是：

```text
MockDataSource 只生成数据
ThingsKitMapper 只拼 JSON
MqttCloudClient 只处理 MQTT
NetManager 只处理网络
CacheStore 只处理缓存文件
GatewayApp 只做主流程编排
```

这个边界目前保持良好。

## 3. 当前阶段最大的风险

### 风险 1：平台物模型与本地 JSON 不是自动完全一致

脚本可以创建产品和设备，也可以在已手动导入过物模型的产品上做刷新、修改或重建。但如果产品是脚本首次创建，ThingsKit 页面侧的物模型仍需要人工导入一次，页面显示才可靠。

应对方式：

- 文档中已经写明该边界。
- 本地 `things_model/*.json` 作为当前源模型。
- 平台侧以人工导入后的页面显示为准。

### 风险 2：命令下发已经能回包，但不是执行成功

当前板端只做命令下发边界：

```text
收到命令 -> 解析 method/params/target -> 校验服务和参数 -> 返回解析结果
```

不做：

```text
真实重启
真实 OTA
真实设置采集周期
真实拉合闸
真实清电量
主动事件上报
```

因此响应中的成功只代表“解析合法”，不代表动作已执行。

### 风险 3：DTU 拓扑和业务设备拓扑不是同一棵树

DTU 拓扑描述通信组网关系，电表拓扑描述业务层级关系。两者有映射，但不能混为一棵树。

当前做法是正确的：

- DTU 用 `topology.parent_mac` 和 `topology.child_macs` 表示通信关系。
- 电表用 `parent_meter_id` 表示业务关系。
- `collect_config.type_1` 表示 DTU 下挂的 Modbus 设备类型。

## 5. 后续阶段建议

后续建议按以下顺序继续，不要一次全部实现：

1. 命令下发联调：只测试平台下发到板端能否被正确解析和响应。
2. 服务执行层：新增独立 `CommandRouter` 或同等轻量模块，把“解析”和“执行”分开。
3. 事件上报：先用测试脚本验证 ThingsKit 事件格式，再接入板端。
4. 真实 DTU/WS73 接入：新增数据源，不改 MQTT 和 Mapper 核心边界。
5. Modbus 真实采集：用真实寄存器替代 mock 值。
