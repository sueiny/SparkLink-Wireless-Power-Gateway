# 边缘网关最终功能与技术栈

> 项目：基于星闪的无线电力透传系统  
> 模块：RK3506 边缘网关 `gatewayd`  
> 平台：RK3506 / Buildroot Linux / ThingsKit  
> 当前日期：2026-05-11

## 1. 网关定位

边缘网关位于星闪 DTU 组网和 ThingsKit 云平台之间，承担现场数据汇聚、统一建模、边缘处理、可靠上云和云端命令接入的职责。

整体链路：

```text
Modbus 设备
  ↓ RS485
DTU 节点
  ↓ 星闪组网
根 DTU / 网关接入口
  ↓
RK3506 边缘网关 gatewayd
  ↓ MQTT / HTTP API
ThingsKit 云平台
```

最终目标是让多个 DTU 节点下挂的电表、温湿度变送器、继电器等设备，经过边缘网关统一汇聚后，在云平台形成可展示、可分析、可告警、可下发控制的完整闭环。

## 2. 最终完整功能

### 2.1 数据接入

最终网关需要支持接收根 DTU 汇聚上来的多类型设备数据。

支持设备类型：

| 类型 | 说明 |
|------|------|
| 单相电表 | 电压、电流、功率、电能、线损、拉合闸状态 |
| 三相电表 | 预留，后续按三相表寄存器扩展 |
| 温湿度变送器 | 温度、湿度、在线状态 |
| 继电器 | 开关状态、控制模式、在线状态 |
| DTU 节点 | 节点角色、MAC、拓扑、采集配置、在线状态 |
| DTU 网关 | 网络状态、云端连接状态、设备数量、缓存数量、版本 |

当前状态：

| 能力 | 状态 |
|------|------|
| 模拟 meter/env/relay/DTU 数据 | 已实现 |
| 真实 DTU 数据接入 | 待接入 |
| 真实 Modbus 解析 | 由 DTU 侧负责，网关侧后续接收统一帧 |
| 多设备统一建模 | 已实现基础模型 |

### 2.2 设备统一建模

网关内部使用统一设备模型承接不同来源的数据。

核心思想：

```text
不同设备原始数据
  ↓
DeviceInfo / TelemetryData
  ↓
ThingsKitMapper
  ↓
ThingsKit 物模型字段
```

当前已对齐的产品物模型：

| 产品 | 物模型文件 |
|------|------------|
| DTU 网关 | `gateway_model.json` |
| 单相电表 | `single_phase_meter_model.json` |
| 温湿度变送器 | `env_sensor_model.json` |
| 继电器 | `relay_device_model.json` |
| DTU 节点 | `dtu_node_model.json` |

DTU 节点重点字段：

| 字段 | 说明 |
|------|------|
| `role` | 0 为根/直连网关，1 为普通节点 |
| `mac` | DTU 节点 MAC |
| `topology` | 父节点、子节点数量、子节点列表 |
| `collect_config` | Modbus 设备数、采集周期、预设地址和类型 |

### 2.3 拓扑管理

系统最终需要同时表达两类拓扑：

| 拓扑 | 含义 | 表达字段 |
|------|------|----------|
| DTU 通信拓扑 | 星闪组网父子关系 | `topology.parent_mac`、`topology.child_macs` |
| 电表业务拓扑 | 总表/支表层级 | `parent_meter_id` |

当前测试拓扑：

```text
dtu网关
├── DTU_001
│   ├── DTU_002
│   │   ├── DTU_004
│   │   └── DTU_005
│   └── DTU_003
│       ├── DTU_006
│       └── DTU_007
├── DTU_008
├── DTU_009
├── DTU_010
└── DTU_011
```

DTU 到下挂设备映射：

```text
DTU_001 ~ DTU_007 -> METER_001 ~ METER_007
DTU_008 ~ DTU_009 -> ENV_001 ~ ENV_002
DTU_010 ~ DTU_011 -> RELAY_001 ~ RELAY_002
```

`collect_config.type_1` 枚举：

| 值 | 类型 |
|----|------|
| 0 | 未定义 |
| 1 | 三相电表 |
| 2 | 单相电表 |
| 3 | 温湿度变送器 |
| 4 | 继电器 |
| 5 | 其他预留 |

### 2.4 云端上报

网关负责将设备数据统一发布到 ThingsKit。

当前 MQTT Topic：

| 数据 | Topic | 状态 |
|------|-------|------|
| 网关属性 | `v1/devices/me/attributes` | 已实现 |
| 网关遥测 | `v1/devices/me/telemetry` | 已实现 |
| 子设备遥测 | `v1/gateway/telemetry` | 已实现 |
| 命令请求 | `v1/devices/me/rpc/request/+` | 已订阅 |
| 网关命令请求 | `v1/gateway/commands/request` | 已订阅 |

子设备遥测格式：

```json
{
  "METER_001": [
    {
      "ts": 1710000000000,
      "values": {
        "voltage": 220.1,
        "current": 3.2,
        "online": true
      }
    }
  ]
}
```

### 2.5 网络接入与选择

最终网关支持三种上云链路：

| 网络 | 接口示例 | 说明 |
|------|----------|------|
| 以太网 | `eth0` / `eth2` | 有线网络 |
| Wi-Fi | `wlan0` | 无线局域网 |
| 4G | `ppp0` | 蜂窝网络 |

支持模式：

| 模式 | 行为 |
|------|------|
| `ethernet` | 只检测以太网 |
| `wifi` | 只检测 Wi-Fi |
| `cellular` | 只检测 4G |
| `auto` | 按优先级选择可用网络 |

自动优先级：

```text
以太网 > Wi-Fi > 4G
```

当前实现：

- 网络接口存在性检测。
- IPv4 地址检测。
- 云端 TCP 连通性检测。
- 网络脚本拉起接口。
- 主循环中断线后重新选择网络。

### 2.6 本地缓存与补传

网关需要在网络异常或 MQTT 发布失败时保存待上报数据，网络恢复后补传。

当前实现：

| 能力 | 状态 |
|------|------|
| JSONL 缓存文件 | 已实现 |
| 失败追加缓存 | 已实现 |
| 恢复后读取补传 | 已实现 |
| 补传成功后重写剩余缓存 | 已实现 |
| SQLite 缓存 | 预留 |

缓存路径：

```text

```

缓存记录包含：


### 2.8 命令下发边界

当前网关已经实现命令下发的解析边界。

已支持：

1. 订阅命令 Topic。
2. 解析 JSON payload。
3. 提取 `method`、`params`、目标设备。
4. 判断目标设备是否存在。
5. 判断服务是否属于当前物模型。
6. 校验基础参数。
7. 返回解析结果。

当前不执行真实动作。

当前保留的服务：

| 设备类型 | 服务 |
|----------|------|
| 网关 | `reboot`、`ota_upgrade` |
| 单相电表 | `set_relay`、`clear_energy` |
| 继电器 | `set_relay`、`set_mode` |
| DTU 节点 | `reboot`、`set_collect_cycle`、`trigger_collect` |
| 温湿度变送器 | 无服务 |

当前响应语义：

```json
{
  "result": true,
  "message": "parsed only, command execution is reserved"
}
```

含义：命令已解析且合法，但没有执行实际控制动作。

## 4. 软件架构

### 4.1 模块划分

```text
gatewayd
├── app         主流程编排
├── config      JSON 配置读取
├── model       设备模型和遥测模型
├── datasource  数据源，当前为 MockDataSource
├── mapper      ThingsKit JSON 映射
├── cloud       MQTT 云端通道
├── network     网络检测和选择
├── cache       失败缓存和补传
├── log         本地日志
└── scripts     网络拉起脚本
```

### 4.2 核心数据流

```text
ConfigManager
  ↓
GatewayApp
  ↓
NetManager.ensureNetwork()
  ↓
MockDataSource.collect()
  ↓
ThingsKitMapper.buildGatewaySubDeviceTelemetryPayload()
  ↓
MqttCloudClient.publishRaw()
  ↓
CacheStore fallback
```

### 4.3 设计原则

| 原则 | 说明 |
|------|------|
| 高内聚 | 每个模块只做自己的事 |
| 低耦合 | 数据源不依赖云端，MQTT 不理解业务模型 |
| 可替换 | 后续可用 `Ws73DataSource` 替换 `MockDataSource` |
| 可调试 | 所有关键流程写日志 |
| 不过度设计 | 第一版不引入复杂框架和无意义抽象 |

## 6. 配置说明

主配置文件：

```text
/userdata/gateway/config/gateway_config.json
```

核心配置项：

| 配置 | 说明 |
|------|------|
| `gateway` | 网关 ID、名称、版本 |
| `thingskit` | MQTT host、port、凭证模式、Topic 前缀 |
| `network` | 网络模式、接口名、优先级 |
| `publish` | 上报周期、网关状态周期、缓存路径 |
| `log` | 日志目录 |
| `devices` | 22 台测试子设备 |

凭证模式：

```json
{
  "credential_mode": "access_token",
  "access_token": "设备 access token",
  "mqtt_basic": {
    "client_id": "旧 MQTT client_id",
    "username": "旧 MQTT username",
    "password": "旧 MQTT password"
  }
}
```

当前默认使用 access token，`mqtt_basic` 仅保留备用。
