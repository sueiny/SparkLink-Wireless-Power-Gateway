# gatewayd 工程约束 v2

> 适用范围：RK3506 第一版网关 `gatewayd`  
> 当前阶段：本地模拟数据 + 以太网 / Wi-Fi / 4G 网络选择 + ThingsKit MQTT 上云  
> 最高优先级：高内聚、低耦合、强可读性、强可维护性、必须多注释、性能兼顾、拒绝过度包装设计

---

## 1. 总原则

本项目第一版代码必须以 **稳定可运行、容易调试、容易扩展** 为目标。

必须遵守：

1. 每个模块只负责一类明确职责。
2. 模块之间通过清晰接口交互，禁止跨层调用。
3. 代码必须容易读、容易改、容易定位问题。
4. 所有关键类、public 函数、核心流程、Linux 特有逻辑必须写注释。
5. 不为了“设计模式”而设计模式。
6. 不为了“面向对象”而拆出过多类。
7. 不做无意义抽象，不写空壳包装。
8. 不牺牲性能去追求形式上的架构漂亮。
9. 第一版优先保证稳定、清晰、可运行、可扩展。
10. 所有配置项必须来自 `gateway_config.json`，禁止硬编码 host、port、路径、接口名。

---

## 2. 高内聚要求

每个模块只做自己的事。

| 模块 | 只允许负责 |
|---|---|
| `MockDataSource` | 生成模拟数据 |
| `ThingsKitMapper` | 内部数据转 ThingsKit JSON |
| `MqttCloudClient` | MQTT 连接和发布 |
| `NetManager` | 网络检测和网络选择 |
| `CacheStore` | 失败缓存和补传 |
| `Logger` | 日志输出 |
| `ConfigManager` | 配置读取 |
| `GatewayApp` | 主流程编排 |

禁止：

```text
MockDataSource 里 publish MQTT
MqttCloudClient 里拼 ThingsKit JSON
NetManager 里处理设备数据
CacheStore 里判断网络状态
GatewayApp 里直接写 Linux 网络检测细节
```

---

## 3. 低耦合要求

必须保证：

1. 数据源不依赖云平台。
2. 云平台不依赖数据来源。
3. 网络模块不依赖业务数据。
4. Mapper 不依赖 MQTT。
5. CacheStore 不依赖 MQTT 和 NetManager。
6. 后续替换 `MockDataSource` 为 `Ws73DataSource` 时，不修改 `CloudClient`、`Mapper`、`NetManager`。
7. 后续替换 JSONL 为 SQLite 时，不修改 `GatewayApp` 主流程。
8. 后续新增 HTTP 上报时，只新增 `HttpCloudClient : ICloudClient`，不推翻 MQTT 结构。

---

## 4. 强可读性要求

代码以“后续自己和队友能快速看懂”为标准。

要求：

1. 函数名表达意图。
2. 变量名不要过度缩写。
3. 单个函数尽量短。
4. 复杂流程拆成私有函数。
5. 不写过深嵌套。
6. 不写一行超长逻辑。
7. 不把多个职责塞进一个函数。
8. 日志信息要能帮助定位问题。

推荐函数名：

```cpp
handleNetworkCycle();
ensureCloudConnected();
publishTelemetryBatch();
flushTelemetryCache();
publishGatewayStatusIfDue();
```

不推荐函数名：

```cpp
runAllLogic();
process();
doWork();
handleEverything();
```

---

## 5. 强可维护性要求

必须为后续扩展留边界，但不能过度设计。

后续扩展方式：

| 后续功能 | 扩展方式 |
|---|---|
| 接 WS73 | 新增 `Ws73DataSource : IDataSource` |
| 接真实 DTU | 新增 `DtuFrameDecoder`，由 `Ws73DataSource` 调用 |
| HTTP 上报 | 新增 `HttpCloudClient : ICloudClient` |
| SQLite 缓存 | 替换 `CacheStore` 内部实现 |
| 规则引擎 | 新增 `RuleEngine` |
| 命令下发 | 新增 `CommandRouter` |
| 批量/压缩 | 新增 `UploadPolicyManager` |

第一版不要提前写：

```text
AbstractFactory
PluginManager
EventBus
IoC Container
复杂状态机框架
多层 Adapter 套 Adapter
PipelineBuilder
FactoryRegistry
```

---

## 6. 多注释要求

必须注释：

1. public 类的职责。
2. public 函数的输入、输出、失败条件。
3. 核心主流程。
4. Linux 专有逻辑，例如 `getifaddrs()`、`/sys/class/net`、非阻塞 `connect()`。
5. MQTT 连接、重连、发布失败处理。
6. 缓存补传逻辑。
7. 为什么某个地方选择简单实现，而不是复杂框架。
8. 重要配置项的含义和默认行为。

推荐注释：

```cpp
// Selects the best available network according to the configured mode
// and priority. This function only checks network reachability.
// It does not connect MQTT or publish telemetry.
NetworkType NetManager::ensureNetwork();
```
但是要求注释用中文！！！！！

不要写无意义注释：

```cpp
// i++
i++;
```

---

## 7. 性能兼顾要求

第一版不是极限性能版本，但不能写明显低效的代码。

要求：

1. 大对象参数优先使用 `const` 引用传递。
2. 循环中避免不必要拷贝。
3. 不在高频循环里重复读取配置文件。
4. 不在每条数据上报时反复初始化 MQTT。
5. 不把缓存文件全量读写放在每一条数据上。
6. 不在主循环里执行长时间阻塞命令。
7. 网络检测必须有超时，不能无限阻塞。
8. JSON 对象用完即释放，不长期保存大量 JSON。
9. 日志不要在高频路径输出过多无效内容。
10. 减少动态分配和重复字符串拼接。
11. 主循环必须可退出，不能写死无限阻塞。

推荐：

```cpp
for (const auto& data : batch) {
    const std::string payload = mapper_.buildTelemetryPayload(data);
    cloud_client_->publishTelemetry(payload);
}
```

不推荐：

```cpp
for (auto data : batch) {  // 会复制 TelemetryData
    ...
}
```

---

## 8. 拒绝过度包装设计

当前项目不是写通用框架，而是写可运行、可维护的网关服务。

禁止为了“架构感”写：

```text
GatewayServiceFactoryManager
AbstractCloudClientFactory
NetworkProviderFactoryRegistry
TelemetryPublishPipelineBuilder
UniversalMessageBus
GenericRuntimeContextContainer
```

判断规则：

> 如果一个接口当前只有一个实现，并且未来短期不会有第二个实现，可以先不用抽象。

可以保留的接口：

| 接口 | 是否合理 | 原因 |
|---|---:|---|
| `IDataSource` | 是 | 第一版 Mock，后续 WS73 |
| `ICloudClient` | 是 | 第一版 MQTT，后续可能 HTTP |
| `INetworkProvider` | 是 | Ethernet / Wi-Fi / Cellular 三个实现 |

暂时不需要的接口：

```text
ILogger
IConfigManager
IThingsKitMapper
ICacheStore
IAppLifecycle
IUploadPolicy
IProtocolCodecFactory
```

## 9. Codex 必须遵守的实现约束

```text
本项目必须严格遵守：
高内聚、低耦合、强可读性、强可维护性、必须多注释、性能兼顾、拒绝过度包装设计。

具体要求：
1. 每个模块只负责一件明确的事。
2. 不允许跨层写逻辑。
3. GatewayApp 只做编排。
4. NetManager 只负责网络选择。
5. MqttCloudClient 只负责 MQTT 连接和发布。
6. ThingsKitMapper 只负责 JSON 映射。
7. CacheStore 只负责缓存文件读写。
8. MockDataSource 只负责模拟数据。
9. public 类、public 函数、核心流程、Linux 特有实现必须写注释。
10. 大对象参数使用 const 引用，避免不必要拷贝。
11. 主循环不能有无超时阻塞操作。
12. 不允许为了设计模式而设计模式。
13. 不要创建没有实际必要的抽象类、Factory、Manager、EventBus、Pipeline。
14. 当前只有一个实现且短期不会替换的模块，不要额外抽接口。
15. 允许保留 IDataSource、ICloudClient、INetworkProvider，因为它们已有明确替换场景。
16. 代码必须以能稳定运行、容易调试、容易扩展为第一目标。
```
