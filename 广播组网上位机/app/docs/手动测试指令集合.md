# 手动测试指令集合

所有命令在 `app/` 目录下执行，串口 `/dev/ttyUSB0`，dongle 地址 `16`（0x10）。

---

## 零、进程管理说明

### CLI 串口守护进程（serial-daemon）

`listen`、`bench`、`send` 等所有需要串口的命令，都通过 Unix socket 连接到 `serial-daemon`，**必须先启动它**，否则报 `ConnectionRefusedError`。

```bash
# 启动串口守护进程（后台运行，保持串口独占）
.venv-d3qn/bin/python -m pc_dijkstra_cli serial-daemon --port /dev/ttyUSB0 &

# 停止守护进程
kill %1          # 或 kill $(pgrep -f "serial-daemon")
```

> **修改路由代码后无需重启守护进程**：`serial-daemon` 只负责串口透传，路由逻辑在 `bench` 等命令自身进程中运行，每次执行命令都加载最新代码。

```bash
# 验证当前 rssi_to_weight 桶是否生效（无需守护进程）
.venv-d3qn/bin/python -c "from pc_dijkstra_cli.routing import rssi_to_weight; print(rssi_to_weight(-103))"
# 新桶应输出 65（旧桶输出 15）
```

### 嵌入式守护进程（gatewayd）

`gatewayd` 运行在硬件侧（RK3506 板），负责 SLE mesh RSSI 采集与 IPC 通信，与上位机 Python 代码**完全独立**。修改上位机代码**不需要**重启守护进程。

仅在以下情况需要重启 gatewayd：
- 修改了 `Gateway/gatewayd/` 下的 C++ 代码并重新烧录
- 设备掉电重启后守护进程未自动拉起

## 一、Dijkstra（sample_dijkstra 模式）

> **说明**：`--route-mode` 已固定为 `sample_dijkstra`，`--send-mode` 已固定为 `single_send`，均可省略。
> 路由采用双张独立表（网关→源、源→目标各自最优），环路路径直接发包，由固件 dedup 处理。

### 标准测试（当前参数）

```bash
# 单阶段（2轮/对，共180轮）——ack-timeout 3.0s（第354次起）
.venv-d3qn/bin/python -m pc_dijkstra_cli bench \
  --port /dev/ttyUSB0 --dongle-addr 16 --rounds 2 \
  --no-retry --ack-timeout 3.0 --interval 0

# 动态模式（2+2轮/对，共360轮，中间自动暂停等 s 继续）
.venv-d3qn/bin/python -m pc_dijkstra_cli bench \
  --port /dev/ttyUSB0 --dongle-addr 16 --rounds 2 \
  --no-retry --ack-timeout 3.0 --interval 0 --dynamic-pause
```

### 参数说明

| 参数 | 值 | 说明 |
|------|-----|------|
| `--rounds` | 2 | 每对轮数（`--dynamic-pause` 时自动 ×2） |
| `--no-retry` | - | 禁用 ACK 超时重试 |
| `--ack-timeout` | 5.0 | ACK 超时秒数 |
| `--interval` | 0 | 轮间 drain 时间（0=不等待） |
| `--dynamic-pause` | - | 跑完第一阶段自动暂停，按 `s` 继续第二阶段（不重采 RSSI） |

### 其他命令

```bash
# 监听串口
.venv-d3qn/bin/python -m pc_dijkstra_cli listen --port /dev/ttyUSB0

# 查看路由（state.json 需存在）
.venv-d3qn/bin/python -m pc_dijkstra_cli routes

# 发送单包
.venv-d3qn/bin/python -m pc_dijkstra_cli send --port /dev/ttyUSB0 --dst 5 --payload AABBCC
```

## 二、单路径手动测试（pc_manual_cli）

> **说明**：手动指定完整路径发包，用于验证特定路径的连通性与延时，不依赖 Dijkstra / D3QN 路由计算。
> 路径中第一个节点为网关（`00`），最后一个节点为目标，中间为指定中继。

### 基本用法

```bash
# 单次测试 01→04，路径经 06 中继
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 01 06 04"

# 多轮测试（10轮，间隔 0.5s，超时 3s）
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 01 06 04" \
  --rounds 10 --interval 0.5 --ack-timeout 3.0

# 测直连路径（00→07→09）
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 07 09" --rounds 5

# 自定义 payload
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 03 08 09" --payload DEADBEEF
```

### 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--port` | 必填 | 串口号 |
| `--baud` | 115200 | 波特率 |
| `--path` | 必填 | 完整路径，空格分隔的十六进制地址，首节点为 `00`，末节点为目标 |
| `--payload` | `AABBCC` | 十六进制 payload |
| `--rounds` | 1 | 发包轮次 |
| `--interval` | 1.0 | 每轮间隔（秒） |
| `--ack-timeout` | 2.0 | 单次 ACK 等待超时（秒） |

### 输出示例

```

指令: SEND 4 4 00 01 06 04 AABBCC
轮次: 5   间隔: 1.0s   超时: 2.0s

[1/5]  发送... ✓  ACK seq=0x0012  251.3ms
[2/5]  发送... ✗  超时 (2.0s)
[3/5]  发送... ✓  ACK seq=0x0013  302.1ms
[4/5]  发送... ✓  ACK seq=0x0014  284.7ms
[5/5]  发送... ✓  ACK seq=0x0015  261.0ms

结果: 4/5 成功  丢包率: 20.0%  平均延时: 274.8ms  最小: 251.3ms  最大: 302.1ms
```

### 常用测试场景

```bash
# 场景1：验证某条绕路路径是否优于直连
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 01 04" --rounds 5        # 直连
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 01 06 04" --rounds 5     # 经 06 中继

# 场景2：验证算法选路是否合理（与 benchmark 路径对比）
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 07 09" --rounds 10       # 算法选路
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 0A 09" --rounds 10       # 手动备选路径

# 场景3：排查特定节点连通性
.venv-d3qn/bin/python -m pc_manual_cli \
  --port /dev/ttyUSB0 --path "00 04" --rounds 3           # 网关直达 04
```

## 三、D3QN

> **说明**：`--send-mode` 已移除，仅保留单次发送模式。

### 基准测试

```bash
# 标准测试（开启在线学习，每 100 轮更新一次）
.venv-d3qn/bin/python -m pc_d3qn_cli bench \
  --port /dev/ttyUSB0 --dongle-addr 16 --rounds 2 \
  --ack-timeout 5.0 --interval 0 \
  --online-interval 100 --online-lr 3e-4 --online-epochs 1 --online-nudge 0.8

# 动态模式（2+2轮/对，共360轮，中间自动暂停等 s 继续）
.venv-d3qn/bin/python -m pc_d3qn_cli bench \
  --port /dev/ttyUSB0 --dongle-addr 16 --rounds 2 \
  --ack-timeout 5.0 --interval 0 \
  --online-interval 100 --online-lr 3e-4 --online-epochs 1 --online-nudge 0.8 \
  --dynamic-pause

# 关闭在线学习（纯推理）
.venv-d3qn/bin/python -m pc_d3qn_cli bench \
  --port /dev/ttyUSB0 --dongle-addr 16 --rounds 2 \
  --ack-timeout 5.0 --interval 0
```

### 在线学习参数

| 参数 | 默认 | 说明 |
|------|------|------|
| `--online-learn` | 开 | 开启测试期在线学习 |
| `--no-online-learn` | - | 关闭在线学习（纯推理） |
| `--online-interval` | 50 | 每多少传输轮触发一次更新 |
| `--online-lr` | 1e-4 | 学习率 |
| `--online-epochs` | 2 | 每次更新的训练轮数 |
| `--online-nudge` | 0.5 | 方向性步长：目标 = 当前Q + 步长×奖励 |

> **注意**：在线学习更新耗时约 1500ms，**计入 `inference_ms`**。
> 真实推理时间约 6~7ms，在线学习触发轮次的 `inference_ms` 会出现尖峰（约 1515ms）。

### 训练

```bash
# 标准训练
.venv-d3qn/bin/python -m pc_d3qn_cli train --iterations 200 --training-episodes 20

# 恢复训练
.venv-d3qn/bin/python -m pc_d3qn_cli train --resume checkpoints/d3qn_mpnn/latest.pt --iterations 200

# 冒烟测试
.venv-d3qn/bin/python -m pc_d3qn_cli train-smoke
```

### 其他命令

```bash

.venv-d3qn/bin/python -m pc_d3qn_cli checkpoint-info

# 预测路径
.venv-d3qn/bin/python -m pc_d3qn_cli predict-path --port /dev/ttyUSB0 --src 0 --dst 5
```

## 六、路由设计说明（最新版）

**两张独立路由表：**
- `gateway_routes`：网关→每个源节点，独立最优，RSSI 采集后预算一次
- `pair_routes`：每对源→目标，独立最优，不受网关路径约束

**环路处理：**
- 合并路径若含重复节点（如 `00→02→01→02→06`），**直接发送不拦截**
- 固件 dedup 表（2s 窗口）决定是否丢包，结果体现为 ACK timeout，计入丢包率
- 不再有 `route_unreachable`（仅 gateway 本身不可达时才出现）

## 七、已知硬件问题

- **节点 04**：能接收但无法回 ACK，所有到 04 的对均为 ACK timeout
- **节点 09**：单向信号，ACK 无法回传，持续 100% 丢包
- **节点 02**：信号不稳定，部分测试中作为目标时丢包率高
- **Dongle 偶尔断开**：需物理拔插 USB，代码自动保存已有结果

## 八、代码变更记录

### routing.py — RSSI 权重分桶（第351次后细化）

旧桶最后一段平坦（所有 ≤-85 dBm 均为 weight=15），导致弱链直连优于多跳中继。当前分桶：

| RSSI 范围 | weight |
|-----------|--------|
| ≥ -55 | 4 |
| ≥ -65 | 5 |
| ≥ -75 | 6 |
| ≥ -85 | 12 |
| ≥ -90 | 18 |
| ≥ -95 | 28 |
| ≥ -100 | 45 |
| ≥ -105 | 65 |
| < -105 | 90 |

### benchmark.py — 报告权重显示修复（第354次后）

报告中「路由跳步 RSSI 详情」表格的权重列由内部 `_bucket()` 函数计算显示，该函数与 `routing.py` 的 `rssi_to_weight()` **相互独立**，第354次前未同步更新，导致表格显示旧桶值（15.0）而实际路由已用新桶。已于第354次后同步修复，两者现保持一致。

验证方式：报告中 RSSI -103/-87 dBm 的边，权重应显示 **41.5**（新桶：(65+18)/2），而非 15.0（旧桶）。

### topology.py — 单向网关边过强问题（待修复）

SAMPLE 模式下，仅有节点单向听到网关的边（网关不上报 RSSI，故反向为 ❌）会触发网关例外逻辑被保留，权重仅取已知方向。若该 RSSI 较弱（如 -99 dBm），Dijkstra 仍可能选此单向弱链直连，导致高丢包。

示例（第354次 06节点）：
- `00→06` 单向 -99 dBm，weight=45 → Dijkstra 选直连 → 100% 丢包
- `00→03→06` 双向均值，cost=64.5 → 实际低丢包但未被选中

**待实施修复**：对单向网关边 RSSI 低于 -95 dBm（weight > 28）的直接丢弃，强制走中继。
