# 关键注意点

## 一、Dongle 操作

### 1.1 Dongle 死机处理

**症状**：测试报错 `SerialException: write failed: [Errno 5] Input/output error` 或 `device disconnected`

**处理**：
```bash
# 检查 dongle 状态
ls /dev/ttyUSB0
dmesg | grep ttyUSB | tail -5

# 如果 /dev/ttyUSB0 不存在，需要物理拔插：
# 1. 拔掉 USB dongle
# 2. 等 3 秒
# 3. 插回去
# 4. 等 5 秒
# 5. 验证：ls /dev/ttyUSB0
```

### 1.2 Dongle 不响应

**症状**：串口打开成功但 RSSI_REQ 无响应

**处理**：
```bash
# 尝试 DTR/RTS 重置
python3 -c "
import serial, time
ser = serial.Serial('/dev/ttyUSB0', 115200, timeout=0.1)
ser.dtr = False; ser.rts = False
time.sleep(0.3)
ser.dtr = True; ser.rts = True
time.sleep(0.3)
ser.dtr = False; ser.rts = False
time.sleep(8)
data = ser.read(8192)
print(f'Got {len(data)} bytes')
ser.close()
"
# 如果仍然无响应，物理拔插
```

### 1.3 不要在代码中做 DTR/RTS toggle

之前尝试在 SerialClient 初始化时做 DTR/RTS toggle，会导致 dongle 断开连接（`/dev/ttyUSB0` 消失）。当前代码只做 `dtr=False, rts=False` + 等待 2 秒 + flush。

## 二、RSSI 拓扑

### 2.1 拓扑波动

每次 RSSI 采集结果不同，导致：
- 路由表不同
- 部分节点可达/不可达
- 测试结果波动

**建议**：
- `--rssi-seconds 20 --rssi-requests 10` 是较稳定的配置
- 不要用 `--rssi-seconds 30`，反而更不稳定

### 2.2 节点4问题

节点4只有网关入边，其他节点无法到达节点4。这是硬件/物理位置问题，无法通过软件解决。

### 2.3 网关无入边

网关（节点0）没有入边是正常的。`topology.py` 中 `graph()` 方法会过滤 `edge.dst == gateway` 的边。ACK 回程由 firmware 处理，不走我们的路由表。

## 三、路由算法

### 3.1 权重函数

文件：`pc_dijkstra_cli/routing.py` → `rssi_to_weight()`

当前权重（已调优）：
```python
def rssi_to_weight(rssi: int) -> Optional[int]:
    if rssi >= -55: return 4   # 强信号（之前是1，太小导致过度偏好）
    if rssi >= -65: return 5   # 较强（之前是3）
    if rssi >= -75: return 6   # 中等
    if rssi >= -85: return 12  # 弱信号
    return None                # 不可用
```

**重要**：之前权重为 1/3/6/12，强信号节点（如节点4，RSSI -61）权重太低（3），导致所有路径都经过它，造成拥塞。改为 4/5/6/12 后分布更均匀。

### 3.2 Hop Penalty

`hop_penalty` 参数在 Dijkstra 计算时每跳累加。用于：
- 限制路径跳数
- 优先选择短路径

当前最优值：`hop_penalty=8`

### 3.3 单次SEND模式

single_send 模式：网关直接发送到目标（经过路由路径）。
- 路径 = `full_path`（网关→源→目标，由 `_resolve_healthy_matrix_routes` 计算）
- 延迟 = SEND→ACK 往返时间

two_send 模式（已弃用）：先发到源，再发到目标。延迟计算有问题。

## 四、常见陷阱

### 4.1 路由不可达不计入丢包率

bench 报告的丢包率 = `(sent - success) / sent`，其中 `sent = 计划轮次 - 路由不可达`。所以如果大量路由不可达，丢包率看起来很低但实际上很多节点不可达。

### 4.2 延迟包含 firmware 层开销

`source_to_target_ms` 包含：
- 上位机构建命令时间
- 串口传输时间
- firmware 发送时间
- 无线传输时间（多次重传）
- firmware ACK 处理时间
- 串口回传时间

最小延迟约 100ms，典型 200-300ms。

### 4.3 测试顺序

Dijkstra 和 D3QN 应该在相近的 RSSI 条件下测试，否则对比不公平。建议：
1. 先跑 Dijkstra
2. 紧接着跑 D3QN（RSSI 条件相近）

### 4.4 日志自动递增

日志目录序号自动递增。查看最新测试：
```bash
ls -t logs/dijkstra_hw/ | head -1
```

## 五、代码修改注意事项

### 5.1 修改权重后要重新测试

权重修改会影响所有路由，需要重新跑多次测试验证。

### 5.2 benchmark.py 的并发问题

`run_benchmark()` 是单线程的，RSSI 采集和 SEND 是串行的。如果修改为并行，需要处理拓扑锁。

### 5.3 topology.py 的 graph() 方法

`graph()` 方法过滤掉了 `edge.dst == gateway` 的边。如果需要构建包含网关入边的图，需要特殊处理。

### 5.4 串口通信是阻塞的

`_send_and_wait_ack()` 在 ACK 超时期间会阻塞。如果 ACK 超时设太长（如 10s），测试会很慢。
