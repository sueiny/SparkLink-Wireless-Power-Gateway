# 星闪广播组网路由系统方案汇报书

## 一、项目背景

本项目面向星闪短距离无线通信场景，设计一套基于广播发现、RSSI 感知、路径计算和逐跳转发的组网路由测试系统。

系统测试环境约为 **100 平方米室内空间**，网络规模为：

| 组成部分 | 数量 | 作用 |
|---|---:|---|
| PC 上位机 | 1 | 路由计算、测试控制、数据展示 |
| dongle 网关 | 1 | 串口与无线网络桥接、无线收发、时间戳记录 |
| 子节点 | 10 | 广播发现、信息上报、临时路径保存、数据转发 |

本系统主要用于验证星闪广播组网能力、节点间多跳通信能力，以及在小规模网络下的时延和丢包率表现。

---

## 二、项目目标

本项目的核心目标包括三部分：

### 1. 实现网络初始化

系统启动后，10 个子节点通过广播方式感知周围节点，记录相互之间的 RSSI 信息，并将节点信息上报至网关。网关将收集到的节点信息透传给 PC 上位机，由上位机汇总形成全局网络拓扑。

### 2. 实现最佳路由计算与下发

PC 上位机根据各节点和网关上报的 RSSI 信息，建立节点间链路关系，并使用 Dijkstra 路由算法计算全局路由表。节点无需保存到网关的固定最佳路由，与网关通信依靠广播和周围节点辅助转发，直到网关收到。

### 3. 实现节点间数据传输与性能测试

当某一节点需要向目标节点发送数据时，源节点向网关请求目标路径，网关通过 PC 查询最佳路由后下发给源节点。源节点按照该路径进行逐跳单播传输，目标节点收到数据后向网关返回 ACK。网关在关键流程中记录时间戳，用于 PC 统计时延和丢包率。

---

## 三、系统总体架构

系统整体采用 **PC 负责计算、网关负责桥接、节点负责通信与转发** 的分层架构。

```text
+----------------------+
|      PC 上位机        |
| 路由计算 / 测试控制   |
| 结果统计 / 数据展示   |
+----------+-----------+
           |
           | 串口通信
           |
+----------+-----------+
|      dongle 网关      |
| 无线收发 / 数据透传   |
| 时间戳记录 / 包过滤   |
+----------+-----------+
           |
           | 星闪广播 / 单播
           |
+----------+-----------+
|        10 个子节点     |
| 广播发现 / RSSI 采集   |
| 临时路径保存 / 逐跳转发 |
+----------------------+
```

## 四、模块职责划分

### 1. PC 上位机

PC 上位机是系统的控制和计算中心，主要负责：

| 功能 | 说明 |
|---|---|
| 节点信息汇总 | 接收网关上传的节点信息包 |
| 网络拓扑构建 | 根据节点 RSSI 信息建立链路图 |
| 路由计算 | 使用 Dijkstra 算法计算最佳路径 |
| 路由表保存 | 保存全局路由表 |
| 路由下发 | 将必要路由通过网关下发给节点 |
| 测试控制 | 指定源节点、目标节点和测试次数 |
| 结果展示 | 显示路径、时延、丢包率等测试结果 |

### 2. dongle 网关

dongle 网关是 PC 与星闪无线网络之间的桥接节点，主要负责：

| 功能 | 说明 |
|---|---|
| 串口通信 | 与 PC 上位机进行数据交互 |
| 无线广播 | 下发初始化、路由、测试等控制信息 |
| 无线接收 | 接收节点信息、路由请求和 ACK |
| 数据透传 | 将节点上报信息转发给 PC |
| 时间戳记录 | 在测试过程中记录关键时间点 |
| 简单过滤 | 对重复包和无效包进行基础过滤 |

### 3. 子节点

子节点是无线网络中的普通通信节点，主要负责：

| 功能 | 说明 |
|---|---|
| 广播发现 | 初始化阶段周期性广播自身信息 |
| RSSI 记录 | 接收周围节点广播并记录 RSSI |
| 信息上报 | 生成并广播节点信息包 |
| 辅助转发 | 帮助网关接收距离较远节点的信息 |
| 临时路径保存 | 按需保存当前通信任务下发的路径 |
| 路由请求 | 需要通信时向网关请求目标路径 |
| 数据转发 | 按照路径进行逐跳单播 |
| ACK 返回 | 目标节点收到数据后返回确认信息 |

## 五、网络初始化方案

网络初始化是系统运行的第一阶段，目的是获取完整网络拓扑并建立基础路由。

### 1. 初始化流程概述

```text
PC 发起初始化
        ↓
网关广播初始化命令
        ↓
所有节点进入发现阶段
        ↓
节点周期性广播自身 HELLO 信息
        ↓
节点记录周围节点 RSSI
        ↓
节点生成节点信息包
        ↓
节点广播节点信息包
        ↓
周围节点辅助转发
        ↓
网关收集节点信息
        ↓
网关透传给 PC
        ↓
PC 汇总拓扑并生成全局路由表
```

### 2. 节点发现机制

初始化开始后，各节点在约 `500 ms` 的发现窗口内周期广播自身信息。其他节点接收到广播后，记录发送节点 ID 及对应 RSSI 值。

例如：

```text
节点 A 收到节点 B 的广播，RSSI = -60 dBm
节点 A 收到节点 C 的广播，RSSI = -72 dBm
```

则节点 A 的本地邻居信息为：

```text
A 能接收到 B
A 能接收到 C
```

这些信息后续将作为 PC 构建链路拓扑的基础。

### 3. 节点信息包

发现阶段结束后，每个节点生成自己的节点信息包，内容包括：

| 内容 | 说明 |
|---|---|
| 节点 ID | 当前节点编号 |
| 初始化轮次 | 区分不同初始化周期 |
| 邻居数量 | 当前节点能接收到的节点数量 |
| 邻居列表 | 邻居节点 ID、RSSI、采样次数等信息 |

节点信息包会通过广播方式发送给网关。对于距离网关较远、无法被网关直接接收到的节点，其信息包可由周围节点辅助转发。

### 4. 广播转发与去重

由于系统采用广播方式传递节点信息，为避免重复转发和广播风暴，系统在每个广播包中加入序号和来源标识。

去重依据为：

```text
原始节点 ID + 命令类型 + 序号 + 初始化轮次
```

同一个数据包只处理一次。若节点收到重复包，则直接丢弃；若收到未处理过的有效包，则加入转发队列并帮助广播。

## 六、路由计算方案

PC 上位机根据网关上传的节点信息建立网络拓扑，并使用 Dijkstra 算法计算最佳路径。

### 1. 拓扑构建原则

节点上报的信息本质上表示“当前节点能听到哪些节点”。

例如：

```text
节点 A 上报：听到节点 B，RSSI = -60 dBm
```

其含义为：

```text
B 的广播可以到达 A
```

因此在链路图中可认为存在一条方向链路：

```text
B → A
```

PC 根据所有节点上报的信息，形成有向链路图。

### 2. 链路权重设计

RSSI 越强，说明链路质量越好，对应的路由代价越低；RSSI 越弱，链路代价越高。

可采用分段方式将 RSSI 转换为路由代价：

| RSSI 范围 | 路由代价 | 链路质量 |
|---:|---:|---|
| `>= -55 dBm` | 1 | 优 |
| `-56 ~ -65 dBm` | 3 | 良 |
| `-66 ~ -75 dBm` | 6 | 中 |
| `-76 ~ -85 dBm` | 12 | 弱 |
| `< -85 dBm` | 不推荐 | 不稳定 |

PC 根据该代价模型计算最优路径。

### 3. 路由表内容

PC 计算完成后，保存全局路由表。路由表主要包括：

| 字段 | 说明 |
|---|---|
| 源节点 | 数据发送方 |
| 目标节点 | 数据接收方 |
| 最佳路径 | 源节点到目标节点的完整路径 |
| 路由代价 | 路径总代价 |
| 路由版本 | 当前路由表版本 |
| 路由状态 | 有效、失效或不可达 |

示例：

```text
A → 网关：A → B → 网关
A → C：A → B → C
C → 网关：C → D → 网关
```

## 七、路由下发方案

初始化结束后，PC 只保存全局路由表，不向节点预置固定网关路径。节点与网关之间的控制信息、路由请求和 ACK 回传仍通过广播和周围节点辅助转发完成，直到网关收到。

当 PC 指定节点间通信或测试任务时，PC 根据当前全局路由表计算本次任务所需路径，并通过网关将该临时路径下发给源节点。

例如，测试节点 A 到节点 C：

```text
节点 A 到节点 C 路径：A → B → C
```

节点 A 按需临时保存：

```text
目标：C
路径：A → B → C
下一跳：B
路由版本：当前版本号
```

本方案中，子节点不保存全局路由表，也不保存固定网关路径，只按需保存当前通信任务所需的临时路径，以降低节点存储和处理复杂度。

节点本地路由保存原则如下：

| 路由类型 | 是否保存 | 说明 |
|---|---|---|
| 本节点到网关路由 | 不保存 | 路由请求和 ACK 上报依靠广播及辅助转发到达网关 |
| 临时目标路由 | 按需保存 | 用于当前节点间通信 |
| 全局路由表 | 不保存 | 由 PC 上位机统一维护 |

## 八、节点间数据传输方案

节点间通信采用“按需请求路由 + 逐跳单播转发”的方式。

### 1. 典型通信场景

假设需要测试：

```text
A → C
```

PC 计算出的最佳路径为：

```text
A → B → C
```

节点 A 初始状态下不保存 A 到 C 的完整路径，因此需要向网关请求目标路由。

### 2. 数据传输流程

```text
PC 指定 A → C 测试
        ↓
网关广播测试指令给 A
        ↓
A 收到测试指令
        ↓
A 向网关请求 A → C 路由
        ↓
网关向 PC 查询最佳路径
        ↓
PC 返回路径 A → B → C
        ↓
网关下发路径给 A
        ↓
A 保存临时路径
        ↓
A 单播数据给 B
        ↓
B 按路径转发给 C
        ↓
C 收到数据
        ↓
C 通过广播和辅助转发方式返回 ACK
        ↓
网关收到 ACK 并上报 PC
```

### 3. 逐跳转发逻辑

数据包中携带完整路径，例如：

```text
path = [A, B, C]
```

各节点根据自身在路径中的位置判断下一跳。

| 当前节点 | 动作 |
|---|---|
| A | 发送给下一跳 B |
| B | 继续转发给下一跳 C |
| C | 判断自己为目标节点，接收数据并返回 ACK |

该方式可以避免中间节点查询路由，简化节点逻辑。

## 九、时延与丢包率测试方案

系统测试阶段由网关统一记录时间戳，节点无需时间同步。

### 1. 时间戳设计

网关在一次 A → C 测试中记录四个关键时间点：

| 时间戳 | 记录时机 | 含义 |
|---|---|---|
| `T1` | 网关发送测试指令给 A | 测试开始 |
| `T2` | 网关收到 A 的路由请求 | A 已收到测试指令并发起请求 |
| `T3` | 网关发送目标路由给 A | 路由查询完成 |
| `T4` | 网关收到 C 的 ACK | 数据传输完成 |

### 2. 时延指标

基于四个时间戳，可计算以下指标：

| 指标 | 计算方式 | 含义 |
|---|---|---|
| 路由请求阶段耗时 | `T2 - T1` | 测试指令下发及路由请求返回耗时 |
| 路由处理阶段耗时 | `T3 - T2` | 网关、PC 查询和路由下发准备耗时 |
| 数据传输阶段耗时 | `T4 - T3` | 路由下发、数据逐跳传输和 ACK 返回耗时 |
| 单次测试总耗时 | `T4 - T1` | 一次完整通信测试总耗时 |

其中，`T4 - T3` 是本方案中最核心的数据阶段测试指标，能够反映多跳链路传输和 ACK 返回的整体性能。

### 3. 丢包率统计

对于多次测试，系统统计发送次数和成功 ACK 次数。

```text
测试总次数 = N
成功收到 ACK 次数 = M
失败次数 = N - M
```

丢包率计算方式为：

```text
丢包率 = (N - M) / N × 100%
```

测试结果可按源节点、目标节点、路径和测试轮次进行统计。

### 4. 失败类型划分

为了便于定位问题，测试失败可分为以下类型：

| 失败类型 | 表现 |
|---|---|
| 测试指令未触达源节点 | 网关未收到源节点路由请求 |
| 路由请求失败 | 源节点请求未到达网关 |
| 路由查询失败 | PC 未找到目标路径 |
| 路由下发失败 | 源节点未收到目标路径 |
| 数据转发失败 | 目标节点未收到数据 |
| ACK 返回失败 | 目标节点收到数据但 ACK 未到达网关 |

## 十、系统工作流程汇总

整体系统可概括为三个阶段。

### 阶段一：网络初始化

```text
节点广播发现
    ↓
节点生成信息包
    ↓
节点广播和转发信息包
    ↓
网关收集信息
    ↓
PC 生成全局路由表
```

### 阶段二：节点间通信

```text
源节点请求目标路由
    ↓
PC 返回最佳路径
    ↓
网关下发路径
    ↓
源节点逐跳单播
    ↓
目标节点返回 ACK
```

### 阶段三：性能测试

```text
网关记录关键时间戳
    ↓
PC 计算各阶段耗时
    ↓
PC 统计成功率和丢包率
    ↓
形成测试结果
```

## 十一、方案特点

### 1. 架构清晰

系统采用 PC、网关、子节点三层结构。PC 负责复杂计算，网关负责桥接和计时，节点负责轻量级通信与转发。

### 2. 节点负担较低

子节点不保存全局路由表，也不保存固定到网关的路径，只按需保存当前通信任务的临时目标路径，降低了节点存储和计算压力。

### 3. 支持远距离节点接入

通过广播转发机制，距离网关较远的节点也可以将自身信息通过周围节点转发至网关。

### 4. 支持多跳数据传输

节点间数据传输采用完整路径逐跳转发方式，可验证星闪网络在多跳场景下的数据传输能力。

### 5. 支持统一时延测试

所有时间戳由网关统一记录，避免节点间时间同步问题，简化测试系统设计。

### 6. 便于后续扩展

后续可在该方案基础上扩展动态路由更新、网关本地路由缓存、多包压力测试、链路质量统计和更复杂的网络拓扑测试。

## 十三、总结

本方案围绕星闪广播组网与多跳路由测试需求，设计了一套由 PC 上位机、dongle 网关和 10 个子节点组成的路由系统。

系统通过节点广播发现和 RSSI 采集建立网络拓扑，由 PC 上位机进行全局路由计算，并将必要路由下发到节点。节点在通信时按需向网关请求目标路径，并按照路径逐跳单播至目标节点。网关在测试过程中统一记录时间戳，PC 根据时间戳完成时延和丢包率统计。

整体方案结构简洁、职责明确，适合用于小规模星闪组网、多跳通信和性能测试验证。
